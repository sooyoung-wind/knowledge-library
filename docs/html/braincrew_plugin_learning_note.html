학습용 노트

브레인크루 플러그인 구조와 운영 방식

발표 이미지에서 추출한 내용을 바탕으로, 브레인크루 플러그인의 핵심 구조를 스킬 레이어, 워크플로우 레이어, 서브 에이전트, 메모리 아키텍처, 조직 지식 축적 전략 중심으로 재구성한 학습 노트입니다.

26개 skill 5개 서브에이전트 9개 훅 4계층 메모리 구조

목차

1. 전체 개요

이 플러그인은 단순한 편의 도구 묶음이 아니라, 팀이 일하는 방식을 구조화하고, 반복되는 시행착오를 조직 자산으로 전환하는 실행 시스템에 가깝습니다.

Skill Layer

우리 스택에 맞는 장비를 선택해 작업 맥락에 따라 자동 로드합니다.

Workflow Layer

인터뷰, 계획, 반박, 구현, 검증까지 표준 실행 루프를 제공합니다.

DIKW Memory Layer

데이터 → 정보 → 지식 → 지혜 → 스킬화의 축적 구조를 만듭니다.

핵심 메시지: 플러그인의 본질은 “좋은 프롬프트 세트”가 아니라, 팀의 작업 방식과 학습 결과를 표준화하고 축적하는 운영 체계입니다.

2. 스킬 레이어

기존 하네스에도 다양한 스킬은 존재하지만, 여기서는 우리 팀에 실제로 필요한 것만 추려 패키징합니다.

스킬 자동 로드 조건
bc-python-conventionPython 작업 시
bc-frontendReact / Next.js 작업 시
bc-langchainLangChain / LangGraph 작업 시
bc-cloud-awsAWS 작업 시 (22개 서비스)
bc-cloud-azureAzure 작업 시
bc-infra인프라 / 백엔드 작업 시
bc-langfuseLangfuse 작업 시
bc-products브레인크루 프로덕트 작업 시

왜 선별이 중요한가

  • 너무 많은 스킬을 동시에 주면 LLM이 어떤 것을 참조해야 할지 혼란스러울 수 있습니다.
  • 필수 스킬 위주로 구성해야 작업 정확도와 일관성이 올라갑니다.

스킬 레이어의 역할

  • 도구를 작업 맥락과 연결합니다.
  • 팀 공통 규칙을 기술적 장비 형태로 내재화합니다.
  • 개발자별 편차를 줄이고 시작 속도를 높입니다.

핵심 설계 원칙: 복잡한 일일수록 계획과 검증 루프를 늘리고, 단순한 일일수록 절차를 압축합니다.

4. 9단계 실행 구조

발표 자료에서 가장 강조된 부분은 “계획 단계의 중요성”입니다. 구현 전에 충분히 묻고, 조사하고, 반박하고, 검증해야 품질이 올라갑니다.

Phase 역할 스킬 / 에이전트
INTERVIEW 철저한 인터뷰를 통해 INTERVIEW-RESULT.md 생성 bc-clarify
RESEARCH 불확실한 부분 조사 (memory, web, arxiv, notion) bc-arxiv, bc-memory-read
PRE-PLAN 가안 수립, 핵심 로직 검증 제안, 병렬 탐색 후보 정리 bc-plan
DEVIL’S ADVOCATE PRE-PLAN을 최소 3가지 관점에서 반박 critic agent
VALIDATE 핵심 로직 사전 검증 (notebook) bc-notebook
PLAN Master Plan + Sub Plans + 동적 도메인 리뷰 bc-plan
DOMAIN REVIEW 도메인별 에이전트 병렬 리뷰 (고정 + 동적) 동적 에이전트 spawn
IMPLEMENT 자율 실행, 불확실할 경우 tree-search 제안 bc-tree-search
PROVE 독립 평가, INTERVIEW-RESULT 기준 정합성 점검 evaluator agent
중요 포인트: 구현은 뒤쪽 단계입니다. 앞단의 인터뷰, 리서치, 계획, 반박이 부실하면 이후 구현 속도가 빨라 보여도 전체 품질은 오히려 떨어집니다.

5. 서브 에이전트

도메인 특성상 Full-stack을 커버해야 하므로, 전문 영역에 특화된 에이전트에게 맥락을 분산(Context Offload)합니다.

Agent 역할 호출 시점
critic Devil’s Advocate. PRE-PLAN에 대해 Risk / Omission / Alternative 관점에서 최소 3가지 반박 DEVIL’S ADVOCATE Phase
evaluator 계획 대비 누락, 코드 품질, 보안, LangChain 컨벤션, FE 테스트 점검 PROVE Phase
lab-runner RAG / Agent 실험 자율 실행 (문서 분석 → 실험 → 리포트) bc-lab
docs-generator README, 다이어그램, HTML 프레젠테이션 생성 /bc-docs
ops-monitor Docker 로그, 컨테이너 진단, 헬스체크 운영 이슈 시

왜 서브 에이전트가 필요한가

  • 한 모델에게 모든 도메인 지식을 동시에 몰아주면 집중력이 분산됩니다.
  • 전문 에이전트가 영역별 검토를 맡으면 품질과 깊이가 올라갑니다.

설계상의 의미

  • 작업을 기능 단위로 쪼개 책임을 분리합니다.
  • 검증과 비판을 독립 모듈로 둬서 자기확신 오류를 줄입니다.

6. 메모리 축적 흐름

메모리는 단순 저장소가 아니라 팀의 일상적 작업 결과를 정리하고, 팀 차원의 자산으로 승격시키는 흐름입니다.

1

CLAUDE.md 규칙

작업 종료 시 공유할 팁이 있으면 tips repo에 PR을 올리도록 유도합니다.

2

개발 세션 종료

팀원 일상 속에서 나온 작은 교훈과 팁을 놓치지 않고 수집합니다.

3

tips repo PR

개별 경험을 문서화 가능한 형태로 변환합니다.

4

주간 Sync

PR들을 주간 단위로 종합해 중복과 패턴을 찾습니다.

5

공유 / 정리

최종적으로 팀 차원의 하네스 자산으로 정리합니다.

실무 팁: “매 작업 시작 시 git pull” 같은 운영 규칙은 사소해 보여도, 메모리 흐름이 끊기지 않게 만드는 핵심 습관입니다.

7. 메모리 레이크 아키텍처

이 구조는 개발 중 발생한 세션 수준의 흔적을 점진적으로 추상화해, 최종적으로는 다시 재사용 가능한 스킬로 환원하는 아키텍처입니다.

단계 1

데이터 (세션 요약)

경로: memory-lake/data/

내용: Haiku가 매 세션 요약

질문: 5개 축 체크

단계 2

정보 (종합 분석)

경로: memory-lake/information/

내용: 5개 세션 데이터 종합

질문: 교훈 발견?

단계 3

지식 (사례 추출)

경로: memory-lake/knowledge/

내용: 문제-시도-해결 케이스

질문: 패턴 반복?

단계 4

지혜 (패턴화)

경로: memory-lake/wisdom/

내용: 문제-추론-솔루션 패턴

질문: 지혜 생성?

단계 5

기술 진화 (적용 실험)

실행 파일: memory_to_skill.py

결과: PR 생성 및 스킬 설치

즉, 메모리는 저장으로 끝나지 않고 다시 실행 가능한 기술 자산으로 환원됩니다.

8. 도입이 어려운 이유

발표 자료는 플러그인을 배포해도 현업에서 강제 도입하기 어려운 현실적 이유를 분명히 짚습니다.

이유 1

작업 스타일의 차이

  • 도구, 프롬프트, 워크플로우 선호가 개인마다 다릅니다.
  • 좋은 규칙이라도 모두가 같은 방식으로 받아들이지 않습니다.
이유 2

맥락과 목적의 차이

  • 업무 성격, 우선순위, 도메인 지식 수준이 다릅니다.
  • 일괄 적용은 오히려 비효율을 만들 수 있습니다.
이유 3

변화에 대한 부담

  • 새 규칙과 도구는 학습 비용을 유발합니다.
  • 강제는 저항을 낳고, 낮은 신뢰도와 저조한 사용률로 이어질 수 있습니다.
시사점: 플러그인의 성공은 “배포 여부”가 아니라, 실제 팀원들이 자기 일에 도움이 된다고 느끼는지에 달려 있습니다.

9. 그래도 반드시 해야 하는 이유

발표 자료의 가장 중요한 메시지 중 하나는 이것입니다. 플러그인 설치의 가치 중 10%는 편의성이고, 90%는 지식 자산 축적입니다.

쉽게 희석된다

프롬프트, 일부 도구 구성은 모델·플랫폼 업데이트로 빠르게 대체될 수 있습니다.

일부 희석된다

유사한 대안이 생기면 기존 자산의 차별성은 약해질 수 있습니다.

희석되지 않는다

조직이 직접 겪은 사례, 실패, 해결 맥락, 고객 맥락은 시간이 갈수록 독점성이 강화됩니다.

조직의 경험 자산이 중요한 이유

  • 이것은 외부 모델 업데이트로 대체되지 않습니다.
  • 팀이 직접 겪은 문제와 해결 맥락이기 때문에 재현성과 설득력이 높습니다.
  • 궁극적으로 조직만의 독점적 경쟁력이 됩니다.

Obsidian KB 문화

백링크 기반의 지식 정리 문화는 팀 지식 구조화의 선행 사례로 읽을 수 있습니다.

Karpathy LLM Wiki

Markdown + grep + RAG 식 접근은 가볍고 실용적인 지식 운영 패턴을 보여줍니다.

Manus File System

Context-offloading은 긴 맥락을 외부 구조로 분산해 처리하는 방향성과 닿아 있습니다.

결론: 앞으로의 LLM 운영은 단순 프롬프트 엔지니어링보다, 백링크·파일 시스템·지식 그래프·메모리 축적을 결합한 “LLM Wiki 패턴”으로 갈 가능성이 높습니다.

10. 한눈에 정리

이 플러그인은 무엇인가

스킬 모음이 아니라, 팀의 작업 방식과 검증 루프와 지식 축적 체계를 묶은 운영 프레임워크입니다.

가장 중요한 단계는

계획 단계입니다. 구현 전에 묻고, 조사하고, 반박하고, 검증하는 구조가 핵심입니다.

서브 에이전트의 역할은

전문 영역별 검토를 맡아 컨텍스트를 분산시키고 품질을 높이는 것입니다.

메모리의 최종 목적은

기록이 아니라 스킬화입니다. 즉, 경험을 다시 실행 가능한 시스템으로 환원하는 것입니다.

도입의 현실적 장벽은

개인별 스타일 차이, 업무 맥락 차이, 변화에 대한 부담입니다.

그래도 해야 하는 이유는

결국 조직의 경험 자산은 희석되지 않는 경쟁력이기 때문입니다.

정리 기준: 업로드된 발표 이미지에서 판독 가능한 내용을 바탕으로 재구성한 학습용 HTML 노트
